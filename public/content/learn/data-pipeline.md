# データパイプライン

## 概要

AI・機械学習を本番運用するには、データを収集・変換・蓄積・提供する**データパイプライン**が不可欠です。ETL・データレイク・ストリーミングといった概念はデータエンジニアリングの基礎です。

---

## ETL（Extract-Transform-Load）

データを**抽出（Extract）→ 変換（Transform）→ 格納（Load）**する一連のプロセス。

| フェーズ | 説明 | 例 |
|---|---|---|
| **Extract** | 各種ソースからデータを収集 | DB・API・ログファイル |
| **Transform** | クレンジング・集計・フォーマット変換 | 日付変換・重複除去 |
| **Load** | データウェアハウスやデータレイクに格納 | BigQuery・Redshift |

**ELT**：TransformをLoad後に行う形式。クラウドDWHの普及で増加。

---

## データアーキテクチャ

### データウェアハウス（DWH）

**構造化データを分析用に最適化した中央集積型DB**。

- 事前にスキーマ定義が必要（Schema-on-Write）
- SQL分析に強い
- 例: Snowflake, BigQuery, Redshift

### データレイク

**あらゆる形式のデータを生のまま蓄積**するストレージ。

- 構造化・非構造化・半構造化すべてを格納
- スキーマ定義は後付け（Schema-on-Read）
- 例: Amazon S3, Azure Data Lake

### データレイクハウス

DWHとデータレイクの統合形態。Delta Lake・Apache Icebergが代表。

## バッチ処理 vs ストリーミング処理

| 比較 | バッチ処理 | ストリーミング処理 |
|---|---|---|
| **タイミング** | 定期的（夜間バッチなど）| リアルタイム |
| **データ量** | 大量を一括処理 | 小量を継続処理 |
| **遅延** | 分〜時間単位 | ミリ秒〜秒単位 |
| **代表ツール** | Spark, Hadoop | Kafka, Flink, Spark Streaming |
| **用途** | 日次レポート・ML学習 | 不正検知・リアルタイム推薦 |

## 主要なデータ処理フレームワーク

| ツール | 種別 | 特徴 |
|---|---|---|
| **Apache Spark** | バッチ+ストリーミング | 大規模分散処理の標準 |
| **Apache Kafka** | メッセージキュー | 高スループットのイベントストリーミング |
| **Apache Airflow** | ワークフロー管理 | パイプラインのオーケストレーション |
| **dbt** | データ変換 | SQL中心のTransformレイヤー |

## データ品質管理

| 観点 | 説明 |
|---|---|
| **完全性** | 欠損がないか |
| **正確性** | データが現実を正しく反映しているか |
| **一貫性** | 異なるシステム間でデータが矛盾していないか |
| **適時性** | データが最新かつ必要なタイミングで利用可能か |

## 試験ポイント

> ⚠️ G検定では以下が頻出です

- **ETLの各フェーズ**の意味（Extract・Transform・Load）
- **データレイク = 生のまま格納**（DWHは構造化）
- **バッチ vs ストリーミング**の違いと用途
- **Apache Kafka = リアルタイムイベントストリーミング**の代表
- **データウェアハウス**はBIツールとの連携に使われる

## 関連用語

| 用語 | 説明 |
|---|---|
| スキーマ | データの構造定義（テーブル・カラム・型）|
| データガバナンス | データの品質・セキュリティ・利用ルールの管理 |
| データカタログ | データ資産のメタデータを管理・検索するツール |
| OLTP | トランザクション処理に特化したDB（業務用）|
| OLAP | 分析処理に特化したDB（DWH用）|
