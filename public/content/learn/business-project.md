# AIプロジェクトマネジメント

## 概要

AIプロジェクトを成功に導くためには、技術的な知識だけでなく、プロジェクト固有のライフサイクル管理・KPI設計・ステークホルダー調整が不可欠。失敗パターンを事前に把握し、PoC段階から本番化・運用改善まで一貫した管理を行うことが求められる。

---

## AIプロジェクトのライフサイクル

```
課題設定 → データ収集・整備 → PoC → パイロット → 本番化 → 運用・改善
```

| フェーズ | 主な活動 | 成果物 |
|---|---|---|
| 課題設定 | 解くべき問題の特定・ビジネス価値の確認 | 課題定義書・As-Is/To-Be分析 |
| データ収集・整備 | データ取得・ラベリング・品質評価 | 学習データセット・EDA報告 |
| PoC（概念実証） | 小規模な実験による実現可能性の確認 | PoC報告・Go/No-Go判断 |
| パイロット | 限定的な環境での実運用検証 | パイロット結果・改善計画 |
| 本番化 | システム統合・インフラ整備・リリース | 本番システム・運用手順書 |
| 運用・改善 | モニタリング・ドリフト対応・継続的改善 | 運用レポート・改善施策 |

---

## PoC（概念実証）の進め方

PoCは「技術的に実現可能か」を短期間・低コストで検証するフェーズ。

**成功のポイント**
- **成功基準を事前に設定する**: 精度・処理速度・コストの目標値を明確化
- **データ品質を早期評価する**: ラベルの品質・量・偏りを確認
- **短期間で完了する**: 2〜4週間を目安。長期化は課題の不明瞭さのサイン
- **Go/No-Go判断を明確にする**: 感情的判断ではなく事前設定基準で決定

**PoCで評価すべき項目**

| 評価軸 | 確認内容 |
|---|---|
| 技術的実現可能性 | 目標精度・速度は達成可能か |
| データ品質 | 必要な量・品質のデータが調達できるか |
| コスト妥当性 | 開発・運用コストに見合うROIが見込めるか |
| 組織適合性 | 現場の業務プロセスに組み込めるか |

## データ収集・ラベリング計画

**必要データ量の見積もり**: タスクの複雑度・モデルの種類・目標精度によって変動。経験則として分類タスクでは1クラスあたり数百〜数千サンプル。

**アノテーション（ラベリング）のコスト管理**
- クラウドソーシング（Amazon Mechanical Turk等）vs 専門家ラベリングのトレードオフ
- ラベリングガイドライン整備によるアノテーターの一致率向上
- Active Learning（不確実なサンプルを優先して人間にラベリング依頼）

**データガバナンス**
- 個人情報・著作権への対応
- データリネージ（データの出所・加工履歴）の管理
- アクセス権限管理

## KPI設定

技術指標とビジネス指標を連動させることが重要。

**技術指標（モデル性能）**

| 指標 | 説明 |
|---|---|
| 精度（Accuracy） | 正解率全体 |
| 適合率・再現率・F1 | 不均衡データや誤検知コスト考慮時 |
| レイテンシ | 推論1件あたりの応答時間 |
| スループット | 単位時間あたりの処理件数 |

**ビジネス指標**

| 指標 | 例 |
|---|---|
| 売上・収益への貢献 | レコメンドによる購買率向上 |
| コスト削減 | 人手作業の自動化による工数削減 |
| 顧客満足度 | NPS・問い合わせ解決率 |
| 業務効率化 | 処理時間の短縮率 |

**連動のポイント**: 技術指標が改善してもビジネス指標が改善しなければ価値がない。KPIツリーを作成し、技術指標→中間指標→最終ビジネス指標の因果関係を明示する。

## AIプロジェクトの失敗パターン

| 失敗パターン | 説明 | 対策 |
|---|---|---|
| データ不足・低品質 | 学習データが少ない・ノイズが多い | データ収集計画を先に立てる |
| 課題設定ミス | 解くべき問題が間違っている | ユーザーインタビュー・AS-IS分析を徹底 |
| PoC貧乏 | PoCを繰り返すだけで本番化できない | Go/No-Go基準と本番化ロードマップを最初から設計 |
| 組織変革の失敗 | 現場がAIを使わない・業務プロセスが変わらない | 変更管理・トレーニング・現場巻き込みを計画 |
| モデルドリフト放置 | 本番後にデータ分布が変わりモデル精度が劣化 | 定期的な再学習・モニタリング体制を構築 |
| 技術先行 | ビジネス課題との紐付けなしに技術実験 | 課題設定→技術選定の順序を守る |

**PoC貧乏（PoC Poverty）とは**: PoCを繰り返して実験の「成功体験」を積み上げるが、組織・コスト・技術的課題が解決されず本番化に至らない状態。

## ステークホルダー管理

| ステークホルダー | 関与事項 |
|---|---|
| 経営層 | 投資判断・KPI承認・組織変革の推進 |
| 現場担当者 | 業務要件定義・データ提供・UI/UX検証 |
| IT部門 | インフラ整備・セキュリティ・システム統合 |
| 法務・コンプライアンス | 個人情報・著作権・規制対応の確認 |
| 外部ベンダー | モデル開発・クラウドインフラ・ラベリング |

## アジャイルAI開発

**特徴**: 短いイテレーション（スプリント）を繰り返し、フィードバックを取り込みながら改善する。

| 概念 | 説明 |
|---|---|
| MVP（Minimum Viable Product） | 最小限の機能で動く製品。早期にユーザーフィードバックを得る |
| フィードバックループ | リリース→測定→学習→改善を素早く回す |
| 反復的改善 | 完成を目指すのではなく、継続的に改善し続ける |

---

> ⚠️ **試験ポイント**
> - **PoC貧乏**: PoCを繰り返すだけで本番化に至らない状態（よく出題される概念）
> - **KPI**: 技術指標（精度・レイテンシ）とビジネス指標（売上・コスト削減）を **連動** させる
> - **失敗パターン**: データ不足・課題設定ミス・モデルドリフト放置・組織変革の失敗が主要4パターン
> - **Go/No-Go判断**: PoCの成功基準は **事前に** 設定しておくことが重要
